HTML

Tables

Tables in HTML have a long history of abuse as a layout element. This is not the intent of the <table> tag; rather, it is intended to present tabular data. Looking again at Rotten Tomatoes, we can see that the content in the aside is tabular data—there are three tables of movies, each listed with a rating, title, and release date. Let's look at the basic code for creating a table. 

<table>
    <tr>
        <td></td>
        <td></td>
        <td></td>
    </tr>
</table>

The primary tag for a table is, as you might guess, <table>. The contents of a table are a bit less obvious. A table in HTML is composed of rows and cells. Rows are represented with <tr>, while cells are represented with <td>, except for header cells, which are represented with <th>. Here is how the aforementioned movies table would be represented in code. 

<table>
    <tr class="movieListing">
        <td class="movieListing-rating">48%</td>
        <td class="movieListing-title">Saw</td>
        <td class="movieListing-date">Oct 31</td>
    </tr>
    <tr class="movieListing">
        <td class="movieListing-rating">85%</td>
        <td class="movieListing-title">John Wick</td>
        <td class="movieListing-date">Oct 24</td>
    </tr>
    <tr class="movieListing">
        <td class="movieListing-rating">41%</td>
        <td class="movieListing-title">23 Blast</td>
        <td class="movieListing-date">Oct 24</td>
    </tr>
    <tr class="movieListing">
        <td class="movieListing-rating">67%</td>
        <td class="movieListing-title">Laggies</td>
        <td class="movieListing-date">Oct 24</td>
    </tr>
    <tr class="movieListing">
        <td class="movieListing-rating">53%</td>
        <td class="movieListing-title">White Bird in a Blizzard</td>
        <td class="movieListing-date">Oct 24</td>
    </tr>
</table>

Tables can be used in more complex ways as well. rowspan and colspan allow you to define a table cell that takes up more than it's space, pushing the other cells out of the way. Here's an example:

This cell has rowspan="2" This cell has colspan="2"
These cells have no rowspan or colspan
This cell has rowspan="3" and colspan="2"

This makes it possible to create varying cell sizes with minimal effort, but it can be a pain to modify, which is the main reason to avoid using tables as a layout tool.

CSS

@import

CSS allows for the inclusion of extra stylesheets using the @import directive. The syntax is as follows: 

@import url(http://fonts.googleapis.com/css?family=Open+Sans:400,700)

This is just one option of including multiple CSS files in a page. In general, it makes more sense to include each stylesheet separately in HTML using <link>. However, neither of these is best practice—the correct thing to do where possible is to only include one CSS file. This is due to the performance cost incurred by transferring multiple files over HTTP.

Floats

As we covered before, floating an element takes it out of the flow of the page, causing subsequent elements to wrap around it. This looks something like this:

This text is going to wrap around the image, which will be floated to the left side using float: left. Any subsequent content will also wrap around the floated element if it can. This can be circumvented in various methods.

If you do not want an element to be wrapped around floated elements preceeding it, you can use a CSS property known as clear. As with float, clear can be set to left or right, or it can be set to both in order to clear all floats. But this introduces dependencies on the order of elements, and it can quickly become unmanageable. To circumvent this, various “clearfixes” have been developed.

A clearfix is a method of clearing floats from a group of elements to prevent them from impacting any subsequent elements. The most concise clearfix today looks this: 

section:before, section:after {
    content: "";
    display: table;
}

section:after {
    clear:both;
}

This example will prevent any floats within a <section> from impacting other parts of the page.

Two-Column Layout

Right now, all of our text flows into a single column. How would we arrange our content into multiple columns? In the olden days of HTML, the answer would have been a table with colspan. Today, there is a better way.

Before we talk about the ideal system for defining columns, let's take a look at an easy way to set up a multi-column layout in CSS. The first thing we need to do is to choose what top-level elements to define as our columns. In this case, we will keep it simple and use the <main> and <aside> elements. We'll add an id to each of them in order to target only the top-level instances of these tags, since HTML5 allows elements such as <article> to contain them as well.

Turning this into a multi-column layout is simple: first we will float both elements to the left, and then we will specify their widths. This will allow them to flow next to each other, while still taking up the full space. Let's see that in code: 

#main, #aside {
    float: left;
}

#aside {
    width: 33%;
}

#main {
    width: 67%;
}

Using percentages rather than pixels here allows the page to remain flexible as we scale our browsers to different widths. In a future lesson, we will discuss how we can change our layout for various screen sizes, but for now, we will assume these column sizes for all browsers. To prevent the layout from getting too wide, we can add a max-width: 1100px to the body that will let the page scale - up to a point.

What would happen if we were to add a padding to one of our columns? In a previous lesson, we discussed the benefits of using box-sizing: border-box, and this is one of those cases. Without that setting, adding a padding to the element will cause our total width to surpass 100%, meaning that our columns will no longer fit next to each other. For example, if we were to define #aside as 

#aside {
    width: 33%;
    padding-right: 2rem;
}

the result would be a column that took up a width of 33% + 2rem, rather than 33%. This would look something like this:

We can see how easy it is to define a quick two-column layout, but when designing a complicated website, this is often insufficient to design everything in an easy-to-maintain way. A common solution to this problem is to use a multi-column grid—most frequently a 12-, 16-, or 24-column grid. This is a great system, but in most cases, it comes at a cost—style information leaking into the HTML. The reason for this is that common implementations of a grid system involve using class names such as .col-xs-4, which effectively means that the number of columns has been defined in HTML, rather than CSS. This is a step up from inline styles, but it still creates a dependency that makes it difficult to change layouts down the road.

There is a better way to do this, and that way is to use a CSS preprocessor such as LESS or SASS. In a few weeks, we will begin to introduce some LESS into our stylesheets, allowing us to create much more flexible and reusable code. For now, we will stick to manually defining our percentages for our layout.
